# Execution / Safety

Companion to `SYSTEM_COMPONENTS.md` - that doc covers ZAO's architecture broadly and is honest about gaps (including, previously, everything below). This one is scoped to the specific comparison the person asked to close:

| | Claude Code | ZAO |
|---|---|---|
| Permission modes | default/acceptEdits/plan/auto/bypassPermissions, interactively switchable | Same five modes, same names, switchable mid-conversation via `usePreferencesStore.setPermissionMode()` (Settings > Permissions). `src/services/execution/permissionModes.js` is the single gate every tool call in `toolOrchestrator.js` goes through - see that file's header for exactly how each mode treats writes vs. terminal commands. One deliberate deviation: ZAO's HARD_BLOCKED terminal tier (`commandSafety.js`) has **no override in any mode**, including `auto`/`bypassPermissions` - a phone agent with real PC shell + device filesystem access has a bigger unattended blast radius than a developer's own terminal, so that floor doesn't move. |
| Checkpoints | Every file edit snapshotted, `Esc Esc` to rewind, independent of git | `checkpointManager.js` + `edit_checkpoints` table. Every `fs_create_file`/`fs_edit_file`/`fs_delete`/`fs_rename`/`fs_move` call snapshots the file's exact prior content (base64, binary-safe) before the mutation happens - works identically with or without a `.git` present, since most of ZAO's SAF-granted folders don't have one. `filesystemTool.rewindToCheckpoint(id)` does the actual restore. No in-app "Esc Esc" gesture yet (there's no keyboard) - restoring is a Settings > Checkpoints list action for now. |
| Hooks (PreToolUse, PostToolUse, SessionStart) | Lifecycle interception for logging/blocking/formatting | `hooksEngine.js` + `hooks` table. Real shell commands, run through the same `pcTerminalTool`/`termuxTerminalTool` the model itself uses. PreToolUse can block a call (output containing `ZAO_HOOK_BLOCK`); PostToolUse runs after with the real result attached, for logging/formatting/notifications; SessionStart fires once from `App.js` after `initDatabase()` succeeds. |
| Subagents / Agent Teams | Isolated-context parallel workers | **Built.** `subagentManager.js` + the `agent_spawn_subagents` tool. `runToolTask()` (`toolOrchestrator.js`) already built a fresh, empty history per call with no outside state read - true isolation was already structurally present, just never exposed as a fan-out primitive. This adds that: the top-level model can spawn 1-5 subagents with self-contained prompts, each runs its own full tool loop concurrently (`Promise.all`), and the parent only ever sees each one's final answer, never its intermediate tool calls. Recursion-guarded to one level - `agent_spawn_subagents` is excluded from a subagent's own tool list. |
| Telemetry / observability (OpenTelemetry events) | OpenTelemetry events | `telemetry.js` + `agent_actions` table - OTel-shaped (trace_id/span_id/parent_span_id/attributes/status/timing), local-first. Every tool call in `runToolTask()`'s loop gets a span, closing the exact gap `SYSTEM_COMPONENTS.md` called out: no queryable record existed of what ZAO did, in what order, with what result. A subagent's spawn call is itself a span, and everything it does nests under it via `parent_span_id`. No real OTel SDK/exporter loop (this is a phone app, not a long-running service) - optional best-effort HTTP forward to a real OTLP collector if `otel_export_endpoint` is set in Settings, off by default. |
| Worktrees | Parallel sessions on different branches | `worktrees.js` + `worktree_sessions` table. Two paths depending on what's reachable: a literal `git worktree add` on the PC terminal backend when there's a real local checkout (full fidelity, PC-only), or a GitHub branch + `forkConversation()` pairing when there isn't (phone-only sessions) - so a parallel branch of work always gets its own isolated chat context, even without a second working directory. |
| Syntax/JSX check before writing | Edit/Write tools won't save unparseable code | `syntaxCheck.js` - a real parser (`@babel/parser`, JSX + TypeScript plugins enabled), not a heuristic. `fs_create_file`/`fs_edit_file` (`filesystemTool.js`) run it automatically on every `.js`/`.jsx`/`.ts`/`.tsx`/`.json` write, BEFORE the write happens - a syntax error fails the call closed (nothing written, no checkpoint taken) instead of saving broken code. Also exposed standalone as `fs_check_syntax` (one file) / `fs_check_project_syntax` (whole folder, recursive, skips `node_modules`/`.git`/`android`/`ios`/`build`/`dist`) for checking files the model didn't just write. |
| Pre-run gate for terminal commands | No direct Claude Code equivalent | `projectRunGate.js`. `terminal_pc_run_command`/`terminal_termux_run_command` (both terminal tools' `runCommand()`, right after `commandSafety.js`) recognize commands that actually start/build/serve a project (`npm start`, `expo start`, `npm run build`, `node x.js`, etc. - see `RUN_PROJECT_PATTERNS`) and run `fs_check_project_syntax` against the whole granted folder first; any failure blocks the command outright (no `confirmed: true` override - a broken project can't be made to run by confirming harder) and hands back exactly which file/line is wrong instead of a crash after the fact. Fails OPEN (never blocks) when there's no SAF folder granted or the scan itself errors - this is a safety net on top of ZAO's own edits, not a hard requirement to run anything. One real caveat: it checks the SAF-granted folder on the phone, which is reliably the right project for `terminal_termux_run_command` (same device) but isn't guaranteed to match byte-for-byte what `terminal_pc_run_command` runs on a separate PC backend. |

## What's genuinely different from Claude Code, and why

- **Plan mode is a hard floor, not a soft one.** In `plan` mode, write/terminal tools are refused outright rather than offered-with-confirmation - there's no in-chat approve/deny UI in the flat tool loop yet (see below), so "ask for confirmation" would currently mean "silently fail the same way," which is worse than being explicit that plan mode is read-only.
- **No live confirmation UI in the flat loop yet.** `planExecutor.js`'s hierarchical plan path already has an approve/reject step UI (`usePlanStore`). `toolOrchestrator.js`'s flat ReAct loop (what most single-message requests actually run through) doesn't - so right now, a tool call that `permissionModes.js` marks `requiresConfirmation: true` fails closed with a clear reason the model relays, rather than actually pausing for a tap. Wiring a real inline confirmation card into `ChatScreen.js` for this path is the natural next step, not yet done here.
- **HARD_BLOCKED has no override anywhere**, on purpose - see the Permission modes row above.

## Where everything lives

```
src/services/execution/
  permissionModes.js    - the 5-mode gate (no execution of its own)
  checkpointManager.js  - records snapshots (restore itself lives in filesystemTool.js)
  hooksEngine.js        - PreToolUse/PostToolUse/SessionStart, runs via existing terminal tools
  telemetry.js          - OTel-shaped spans -> agent_actions
  subagentManager.js    - fan-out/fan-in over runToolTask()
  worktrees.js          - git worktree (PC) or branch+fork (phone)
  syntaxCheck.js         - real parser (@babel/parser) - JS/JSX/TS/TSX/JSON validity
  projectRunGate.js      - blocks a "start/build" terminal command when syntaxCheck fails
```

New tables (`src/db/database.js`): `edit_checkpoints`, `hooks`, `agent_actions`, `worktree_sessions`. New `user_preferences` columns: `permission_mode`, `otel_export_endpoint`.
